# PX4 无人机导航与避障策略

本文档面向本仓库的下一阶段目标：在现有 OAK-D 感知、IMU 融合与 PX4 消息基础上，逐步形成可落地的无人机导航与避障系统。

当前仓库已经具备的底座：

- `oakd_perception`：发布原始 IMU 与深度点云。
- `imu_fusion`：输出融合姿态并广播 TF。
- `px4_msgs`：提供 PX4 对接所需的 ROS 2 消息定义。
- `px4_comm_bridge`（可执行 `px4_bridge_node`）：PX4 外部控制与数据桥接入口。
- `uav_bringup`：预留系统编排入口。

---

## 目标定义

PX4 侧最终要解决的不是“看见障碍物”，而是以下三件事：

1. 持续感知前方与周边空间占用。
2. 在可飞行空间内生成安全轨迹或速度指令。
3. 将指令稳定送入 PX4 Offboard/导航链路，并在异常时安全降级。

因此，本项目的导航层建议分为两条路线：

- `nav` 导航方案：以二维或 2.5D 代价地图为核心。
- `3D` 导航方案：以点云、体素或 ESDF 为核心。

---

## 方案对比

| 维度 | nav 导航方案 | 3D 导航方案 |
|------|-------------|------------|
| 空间表达 | 2D / 2.5D 占据栅格 | 3D 点云 / 体素 / ESDF |
| 典型输入 | 深度投影后的平面障碍物 | 原始点云、体素地图、局部表面 |
| 适配场景 | 结构化室内、平面地形、快速验证 | 室内外混合、复杂高度层、非平面障碍 |
| 计算开销 | 低到中 | 中到高 |
| 调参难度 | 低 | 高 |
| 对无人机适配度 | 只能做局部简化，容易丢失高度信息 | 更符合 UAV 的真实飞行空间 |
| 对 overhang / 电线 / 树冠 | 较弱 | 较强 |
| 落地速度 | 快 | 慢 |
| 长期可扩展性 | 一般 | 高 |

### 结论

- 如果目标是尽快做出“能飞、能躲”的原型，优先走 `nav` 导航方案。
- 如果目标是长期演进到 PX4 无人机自主避障，最终应收敛到 `3D` 导航方案。
- 最稳妥的路线是：`nav` 先做安全壳，`3D` 做主导航。

---

## nav 导航方案

这里的 `nav` 不是照搬地面车导航，而是把 3D 感知压缩成适合 PX4 Offboard 的二维局部导航层。

### 处理链路

```text
/oakd/points
  → 地面/高度过滤
  → 2D 局部代价地图
  → 局部路径规划 / 避障决策
  → PX4 Offboard 速度或位置指令
```

### 常见实现方式

- 将点云投影到水平面，生成局部 occupancy grid。
- 按高度阈值过滤地面、天花板和过高区域。
- 使用局部路径规划器输出速度向量或短距离目标点。
- 通过 `px4_comm_bridge`（`px4_bridge_node`）发送位置/速度 setpoint 给 PX4。

### 优点

- 计算量低，适合边缘设备先跑通。
- 逻辑清晰，容易调试和回放。
- 可作为安全层，快速屏蔽近距离障碍。

### 缺点

- 丢失高度层信息，容易误判可飞空间。
- 对树冠、桥下、电线、斜坡等场景不可靠。
- 更像“局部避障”，不是完整的三维自主导航。

### 适合的阶段

- 原型验证。
- PX4 Offboard 通信链路打通。
- 低风险场景的局部避障。

---

## 3D 导航方案

3D 导航把无人机周围空间按真实体积建模，更符合飞行器的运动特性。

### 处理链路

```text
/oakd/points
  → 去噪 / 下采样 / FOV 过滤
  → 体素地图 / 局部点云地图 / ESDF
  → 3D 轨迹规划或速度规划
  → PX4 Offboard 指令
```

### 常见实现方式

- 构建局部 3D 体素地图，标记占用与自由空间。
- 基于 ESDF 计算离障碍最近距离，用于安全走廊或优化控制。
- 输出三维速度指令、姿态指令或局部轨迹点。
- 结合 IMU 姿态和 TF，保证点云与机体坐标一致。

### 优点

- 真实反映 UAV 的飞行空间。
- 可处理多高度层障碍与复杂环境。
- 更适合后续扩展到自主穿越、绕障、局部重规划。

### 缺点

- 算法、地图和参数复杂度明显更高。
- 对算力、同步、点云质量更敏感。
- 调试成本高于二维方案。

### 适合的阶段

- 中长期目标。
- 室内复杂环境、树林、廊道、障碍密集区域。
- 需要较高自主性的 PX4 机载导航。

---

## 推荐总体路线

### 阶段 1：感知底座稳定

- 稳定输出 `/oakd/imu/raw` 与 `/oakd/points`。
- 确保 IMU 融合、TF 广播与 RViz 可视化正常。
- 统一消息时序与机体系坐标。

### 阶段 2：PX4 Offboard 打通

- 在 `px4_comm_bridge` 中实现最小闭环。
- 先实现解锁、起飞、定点悬停、速度控制。
- 建立安全降级逻辑，例如点云丢失时悬停或返航。

### 阶段 3：nav 安全壳

- 以 2D / 2.5D 局部地图做快速避障。
- 优先覆盖“前方近距离障碍物拦截”场景。
- 用于快速验证整条链路。

### 阶段 4：3D 主导航

- 引入 3D 局部地图或 ESDF。
- 支持多层障碍、复杂穿行和局部重规划。
- 将 3D 导航作为最终主方案，nav 作为回退层。

---

## 本仓库内的建议分工

### `oakd_perception`

- 继续负责 IMU 原始数据、深度点云和 FOV 过滤。
- 为导航层提供稳定、可配置、低噪声的数据源。

### `imu_fusion`

- 继续提供姿态融合与 TF。
- 确保导航层拿到统一坐标系。

### `px4_comm_bridge`

- 作为 PX4 指令出口。
- 后续承载位置控制、速度控制、避障接管和安全降级。

### `uav_bringup`

- 作为系统编排入口。
- 后续整合感知、融合、导航和 PX4 通信。

---

## 和 PX4 的接口建议

建议优先围绕以下能力设计：

- Offboard 位置/速度控制。
- 失联保护与安全悬停。
- 基于点云的局部障碍物约束。
- 必要时使用 PX4 自带的路径规划或避障命令作为补充。

从仓库现状看，`px4_msgs` 已经提供消息层基础，后续重点是把导航层输出稳定映射到 PX4 可执行的 setpoint 和安全状态机。

---

## 结论

如果你问“应该先做哪条路”，答案是：

1. 先做 `nav` 方案，尽快跑通 PX4 Offboard + 局部避障闭环。
2. 同步准备 `3D` 方案的数据结构与坐标系。
3. 最终以 `3D` 导航作为主方案，`nav` 作为安全兜底和快速模式。

这样既能保证项目进度，又不会把系统锁死在二维简化模型里。
